1. Executive Summary
RefCheck is a purpose-built SaaS platform for automating the reference checking process for New Zealand
and Australian recruitment agencies and employers. It replaces manual email chasing and PDF form
distribution with a smart, mobile-first workflow: employers create a candidate record, the system tokenises
and dispatches questionnaires to referees via email and SMS, referees complete a structured form on any
device without logging in, and a consolidated report is generated automatically.
The platform is scoped exclusively to reference checking — no background screening, no visa monitoring,
no psychometric testing. It is designed from the ground up for the NZ/AU market, with a full-featured
questionnaire and form builder that lets employers create, manage, and deploy all three form types
(questionnaire, candidate form, referee form) from a single interface, pre-built question templates for
regulated roles (ECE/childcare, healthcare, trades), fraud-detection heuristics appropriate for that
regulatory context, and integration with FLOCC CRM as the initial third-party connector.
This document is the full technical handoff for the development team. It covers the Airtable data schema, 10
sprint plans with user stories, technical tasks, and acceptance criteria, plus deployment architecture on
Render and GitHub. Sprint 2 — the Questionnaire and Form Builder — is the most strategically important
sprint after the foundation: it delivers the full drag-and-drop builder UI, all 8 question types, conditional
branching logic, risk flagging, an AI question generator, and 7 pre-built industry templates. Each sprint is
two weeks in duration; the full product build is estimated at 20 weeks (~5 months) from kick-off to
production launch.

2. User Roles
Employer / Recruiter
 AUTHENTICATED
  •  Creates candidate records and initiates reference check requests
  •  Selects question templates and packages
  •  Views live status of all checks — per candidate and per referee
  •  Receives fraud flag alerts and turnaround delay notifications
  •  Exports PDF reports for internal use or client delivery
  •  Manages team members and role assignments
Candidate
 TOKENISED LINK — NO ACCOUNT
  •  Receives a one-time tokenised email from the employer
  •  Submits referee details via a mobile-friendly form (name, email, phone, relationship, employer, job title, dates)
  •  Can add substitute referees if prompted
  •  No platform account required — interaction is entirely link-based
Referee
 TOKENISED LINK — NO ACCOUNT
  •  Receives a tokenised email + SMS with a direct link to their questionnaire
  •  Completes a multi-step, mobile-optimised form (no login required)
  •  Receives auto-nudge reminders if response is overdue
  •  Responses are stored directly in Airtable on submission
Admin
 PLATFORM-LEVEL ACCESS
  •  Full visibility across all employer accounts on the platform
  •  Can manage employer subscriptions, user accounts, and templates
  •  Access to platform-wide analytics and audit logs
  •  Reserved for Wayne Sullivan's internal team (Candidex staff)

4. Sprint Plan
10 × 2-week sprints · 20 weeks total
Each sprint is two calendar weeks. Sprints should be run sequentially — later sprints have hard
dependencies on earlier ones. Effort ratings: S = 1–3 days, M = 3–5 days, L = 1–1.5 weeks, XL = 2 full weeks.

XL
GOAL
Establish the full technical skeleton: Airtable base, Node.js API on Render, React app scaffold, Google SSO
login, JWT session management, and employer registration flow. By end of sprint, an employer can sign in
and see an empty dashboard.
USER STORIES

SPRINT
2
Questionnaire / Form Builder
XL
GOAL
Build the unified form builder that lets employers create, manage, and deploy three interconnected form
types from a single interface: the Questionnaire (referee-facing questions), the Candidate Form (referee
nomination and consent), and the Referee Form (the rendered questionnaire per referee). The builder
supports 8 question types, per-question conditional logic, risk flagging, block-on-condition rules,
drag-and-drop reordering, a 7-template pre-built library, and an AI question generator powered by GPT-4o.
This is the most important sprint after the foundation — everything else in the product depends on
questionnaires existing before they can be deployed.
FORM TYPES
•  Questionnaire — The set of questions referees answer. The core of every reference check.
•  Candidate Form — What the candidate fills in before referees are contacted: referee nomination,
consent, relationship details.
•  Referee Form — The referee-facing version of the questionnaire, adapted per conditional logic and
branching rules.

PER-QUESTION SETTINGS
•  Label — The question text displayed to the referee.
•  Description — Optional helper text shown beneath the label.
•  Required / Optional — Toggle — prevents form submission if blank when required.
•  Character limit — For text fields: min and max character count.
•  Risk flag rule — If answer matches condition, auto-flag for employer review.
•  Branching rule — If answer matches condition, show or hide another question.
•  Block rule — If answer matches condition, block referee submission with a custom message.
CONDITIONAL LOGIC / BRANCHING ENGINE
Each question can have one or more if/then rules. Rule format: IF [question X] [operator] [value]
THEN [action] [target]. Operators: equals, does not equal, contains, is greater than, is
less than. Actions: show question, hide question, flag as risk, block submission.
•  Rules are evaluated client-side in real time as referees fill in the form

AI QUESTION GENERATOR

pre-assigned. The employer can accept all, edit individual questions, or discard. Questions are inserted into
the current questionnaire being edited.

FLOCC Integration & Webhooks
L
GOAL
Build the FLOCC CRM integration: incoming webhook triggers automatic reference check creation when a
candidate reaches 'Offer Stage' in FLOCC; outgoing webhook posts completed results back to the FLOCC
candidate record. Also build an employer-configurable outbound webhook system and a basic API
documentation page.
USER STORIES
→  As a Employer using FLOCC, I want to have a reference check automatically triggered when I move a
candidate to Offer Stage so that I never forget to initiate a reference check.
→  As a Employer using FLOCC, I want to see completed reference results appear back in the FLOCC
candidate record without manual data entry so that my workflow stays in one place.
→  As a Any Employer, I want to configure a webhook URL to receive check completion events in my own
systems so that I can integrate RefCheck into my existing tooling.
→  As a Developer, I want to access clear API documentation so that I can build integrations without guessing.
TECHNICAL TASKS
•  FLOCC inbound webhook
    o  POST /webhooks/flocc/inbound — endpoint secured with HMAC signature verification
    o  Payload: { candidateId, candidateName, candidateEmail, candidatePhone, role, employerName,
flocCandidateRecordId }
    o  On receipt: validate signature → look up employer by apiKey (sent as header) → create Candidate record →
assign default package → dispatch candidate invite email
    o  Store flocCandidateRecordId on Candidate for results post-back
    o  Return 200 with { referenceCheckId } immediately — process async
•  FLOCC outbound webhook
    o  Trigger: when candidate overallStatus → 'Complete' or 'Flagged'
    o  POST to FLOCC webhook URL (configured per employer)
    o  Payload: { flocCandidateRecordId, status, overallRating, riskScore, flagCount, pdfReportUrl, completedAt }
    o  pdfReportUrl: generate signed time-limited URL to PDF (24h expiry)
    o  Retry logic: 3 retries with 5m, 15m, 30m backoff
    o  Log delivery attempts in a Webhook Logs table in Airtable
•  Employer outbound webhook builder
    o  Settings page: 'Webhook' tab — URL field, toggle active/inactive, test button
    o  POST /employers/:id/webhook — save webhook URL to Employer record
    o  POST /webhooks/test — sends test payload to configured URL, shows response
    o  Webhook fires on: check complete, check flagged, referee non-response Day 6 alert
•  API key management
    o  Generate apiKey on Employer creation (crypto.randomBytes(32).toString('hex'))
    o  Settings page: 'API Keys' tab — display masked key, Regenerate button
    o  PATCH /employers/:id/api-key — regenerates key (invalidates old key immediately)
    o  All inbound webhook endpoints validated via apiKey in Authorization: Bearer header
•  API documentation page
    o  Static page at /docs (React route, publicly accessible)
    o  Sections: Authentication (Bearer token), Endpoints (GET /candidates, POST /candidates, POST
/webhooks/flocc/inbound, webhook payload schemas), Error Codes (400, 401, 403, 404, 422, 429, 500)
    o  Built with a simple React component — no third-party docs framework needed at this stage

6.4 GitHub Actions CI/CD
Recommended GitHub Actions workflow for the API service:
name: Deploy API
on:
push:
branches: [main]
paths: ['api/**']
jobs:
deploy:
runs-on: ubuntu-latest
steps:
- uses: actions/checkout@v4
- name: Deploy to Render
run: curl -X POST ${{ secrets.RENDER_DEPLOY_HOOK }}
Store RENDER_DEPLOY_HOOK as a GitHub Actions secret (not a repo variable). Render provides a unique
deploy hook URL per service under Settings → Deploy Hooks. Separate deploy hooks for refcheck-api
and refcheck-web.
